브랜드 의사결정 프로토타입에서 범용 Human-AI Decision Intelligence Engine으로 발전하기 위한 현재 위치와 다음 과제를 정리한 보고서

작성 기준: 2026년 6월 29일 · 코드 정적 분석 및 프로젝트 문서 기준

1. Executive Summary

Project Origin이라는 브랜드 의사결정 프로토타입은 약 60~70% 구현됐으며, 최종 목표인 범용 Decision Intelligence Engine 기준으로는 약 20~30% 단계다.

Project Origin은 단순한 AI 이름 생성기가 아니다. 창업자의 비정형 답변을 구조화하고, 의미와 전략을 분석한 뒤, 자체 엔진이 후보를 생성·필터링·평가하고 LLM이 결과를 설명하도록 설계된 Human-AI 의사결정 시스템이다.

현재는 브랜드 분야에서 전체 파이프라인의 골격이 구현됐고, 성공 브랜드를 분석해 생성 지식으로 바꾸는 연구 파이프라인도 구축됐다. 최근에는 연구 결과를 읽어 GenerationRules로 변환하는 단계까지 연결됐다. 다만 생성 규칙이 실제 NamingGenerator와 평가기에 적용되는 마지막 통합, 객관적인 외부 평가, 사용자 결과 학습은 아직 남아 있다.

2. 최종 목표와 현재 제품의 관계

장기 미션 신뢰할 수 있는 AI로 인간의 판단 능력을 향상한다 ↓ Project Origin 브랜드 전략과 네이밍 의사결정에서 구조를 검증한다 ↓ Decision Data 입력·후보·평가·선택·결과를 축적한다 ↓ Origin Engine 검증된 지식·추론·평가·보고 계층을 추출한다 ↓ ReconOS 및 미래 제품 보안과 다양한 산업의 의사결정을 지원한다

따라서 현재 프로젝트의 직접적인 목표는 브랜드 전략 보고서를 판매할 수 있는 Concierge MVP를 만드는 것이다. 장기 목표는 이 과정에서 검증된 구조를 범용 엔진으로 발전시키는 것이다. 지금은 범용 엔진을 완성하는 단계가 아니라, 첫 번째 도메인에서 핵심 구성요소를 실험하고 증명하는 단계다.

4. 현재 구현된 핵심 능력

4.1 비정형 정보를 의사결정 맥락으로 구조화

창업자의 문제, 고객, 비전, 원칙, 차별점을 FounderProfile로 변환하고, 의미 테마와 어휘를 가진 SemanticProfile, 네이밍 방향을 가진 BrandLanguage로 발전시킨다. 이는 범용 Decision Intelligence의 Context Modeling에 해당한다.

4.2 도메인 지식 기반 선택지 생성

브랜드 언어를 형태소 풀과 접미사, 음성 규칙에 연결해 이름 후보를 생성한다. LLM에 모든 생성 책임을 맡기지 않고 생성 과정을 자체 통제한다는 점이 핵심이다.

4.3 제약조건 필터링과 다기준 평가

중복, 길이, 예약어, 발음 가능성을 필터링하고 발음, 독창성, 전략 적합성, 기억성에 따라 후보를 평가·랭킹한다. 이는 범용 엔진에서 Alternative Generation, Constraint Filtering, Multi-Criteria Evaluation에 해당한다.

4.4 LLM 기반 설명과 보고서

상위 후보와 창업자 맥락을 LLM에 전달하고, 엄격한 JSON 스키마로 전략 보고서를 생성한다. 응답은 Validator와 Parser를 거쳐 Markdown 보고서로 렌더링된다. 목표는 AI가 결정을 독점하는 것이 아니라 인간이 판단할 수 있는 근거를 제공하는 것이다.

4.5 브랜드 연구를 생성 지식으로 변환

BrandCollector → BrandAnalyzer → BrandGenomeNormalizer → BrandGenomeValidator → BrandGenomeBuilder → PatternExtractor → KnowledgeCompiler → NamingKnowledgeLoader → GenerationRulesBuilder

이 파이프라인은 성공 브랜드의 언어적·심리적 특성을 구조화하고 통계 패턴을 계산한 뒤, 생성기가 사용할 수 있는 규칙으로 컴파일한다. 구조적 완성도는 높아졌지만 현재 Brand Genome의 표본은 5개이므로 통계적 결론은 아직 실험 수준이다.

5. 현재 끊겨 있는 연결

5.1 GenerationRules가 생성기에 적용되지 않음

naming_knowledge.json → NamingKnowledgeLoader → GenerationRulesBuilder → GenerationRules ✕ NamingGenerator / NameEvaluator

연구 결과를 규칙 객체로 만드는 단계까지는 구현됐다. 그러나 생성기는 아직 목표 길이, 음절, 모음 비율, 자음 성향, 선호 스타일을 입력받지 않는다. 이 연결이 완료돼야 “연구 데이터가 생성 품질을 개선한다”고 말할 수 있다.

5.2 필터링 결과가 평가에 사용되지 않음

애플리케이션은 filtered_names를 계산하지만 평가기에는 필터 이전의 names를 전달한다. 데이터 흐름을 닫기 위해 필터 결과를 평가와 랭킹에 전달해야 한다.

5.3 BrandKnowledge가 후속 판단에 충분히 연결되지 않음

KnowledgeBuilder가 구조화된 브랜드 지식을 만들지만 현재는 디버그 출력 외에 프롬프트, 생성기, 평가기에서 직접 사용하지 않는다.

6. 최종 비전에 필요한 미구현 요소

범용 Decision Intelligence Engine에는 다음 공통 모델이 필요하다.

DecisionContext ├─ objective ├─ stakeholders ├─ constraints ├─ evidence ├─ assumptions ├─ alternatives ├─ criteria / weights ├─ scores / uncertainty ├─ risks ├─ recommendation ├─ human_decision └─ outcome
  • 불확실성: 점수와 확신도를 분리하고 근거 품질을 표현해야 한다.
  • 증거 추적: 추천이 어떤 입력과 출처에서 파생됐는지 연결해야 한다.
  • 가정 관리: 사실, 추론, 가설을 구분해야 한다.
  • 인과 검증: 성공 사례의 상관관계를 성공 원인으로 오해하지 않아야 한다.
  • 결과 학습: 추천, 인간의 선택, 실제 결과를 저장하고 평가 기준을 개선해야 한다.
  • 다중 도메인: 두 번째 제품에서 공통 구조가 반복되는지 확인해야 한다.

7. 테스트와 품질 검증

현재 자동 테스트는 Semantic Engine, Report Validator, Report Parser를 중심으로 한다. Knowledge Loader와 Generation Rules용 파일은 현재 main() 출력 확인 스크립트이며 assertion 기반 pytest 테스트는 아니다.

추가로 필요한 검증은 다음과 같다.

  • Generator, Phonetic Rules, Filters, Evaluator, Ranker 단위 테스트
  • Genome Builder, Pattern Extractor, Knowledge Compiler 테스트
  • Knowledge → Rules → Generator 통합 테스트
  • 인터뷰부터 보고서까지 End-to-End 테스트
  • 일반 ChatGPT 결과와 Project Origin 결과의 블라인드 비교
  • 브랜드 전문가와 실제 창업자의 독립 평가
  • 최종 선택률, 탈락 이유, 사후 만족도 기록
핵심 주의점: 현재 Benchmark는 엔진이 생성한 결과를 엔진 자신의 규칙으로 다시 평가한다. 내부 회귀 검증에는 유용하지만 실제 사람이 더 좋은 이름으로 인식하는지 증명하지는 못한다.

8. 권장 우선순위

  1. GenerationRules를 NamingGenerator와 평가기에 실제 연결한다.
  2. 목표 길이, 모음 비율, 자음 성향과 스타일을 생성·필터·점수에 반영한다.
  3. filtered_names가 평가 단계로 전달되도록 애플리케이션 흐름을 수정한다.
  4. Knowledge Loader와 Generation Rules를 실제 pytest 테스트로 전환한다.
  5. 산업 분류 체계를 정규화하고 Brand Genome을 산업별 최소 수십 개 이상으로 확대한다.
  6. LLM 분석을 전문가 또는 외부 데이터로 교차 검증한다.
  7. 일반 LLM과의 블라인드 비교로 제품 품질을 측정한다.
  8. 고객의 선택, 탈락 이유, 만족도와 결과를 저장하는 Decision Outcome을 도입한다.
  9. 브랜드 제품의 유료 검증 후 두 번째 도메인에서 공통 구조를 확인한다.
  10. 두 제품에서 실제로 반복된 계층만 Origin Engine으로 추출한다.

9. 최종 평가

Project Origin은 아이디어만 있는 단계는 지났다. 브랜드 의사결정 제품의 핵심 프로토타입과 연구 파이프라인이 존재하고, 연구 결과를 생성 규칙으로 바꾸는 통합 단계에 진입했다.

브랜드 MVP 기준으로는 중반을 넘었고, 범용 AI 의사결정 엔진이라는 최종 목표 기준으로는 기반 공사를 마친 초기 단계다. 다음 성장은 모듈 수보다 연결 완성도, 데이터 신뢰도, 외부 품질 검증, 실제 사용자 결과에서 결정된다.

현재 위치: “브랜드 분야의 Explainable Decision Prototype”을 구축했고, 이를 “데이터로 개선되는 Human-AI Decision Engine”으로 전환하는 단계.

본 평가는 저장소의 코드와 프로젝트 문서를 기준으로 작성한 개발 현황 분석이며, 비율은 제품 가치와 기술 성숙도를 설명하기 위한 추정치다.
